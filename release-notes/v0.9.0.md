## v0.9.0 — 存档监控与诊断

v0.9 为 BAS 补上了诊断能力：一个 Prometheus 指标接口和一条区块耗时排行命令，让服主不用外挂 profiler 就能看到存档性能的全貌。

### Prometheus 指标接口

开启后，BAS 会在指定端口提供一个 `/metrics` 页面，输出 Prometheus 标准格式的运行数据——区块保存次数、各阶段耗时分布、队列深度、失败计数等。接入 Grafana 就可以画出长期趋势图，比逐次执行 `/betterautosave debug` 看快照方便得多。

默认关闭，在 `common.toml` 中启用：

```toml
[prometheus]
enabled = true
# bindAddress = "0.0.0.0"  # 监听地址，默认对所有接口开放
# port = 9450              # 端口
```

Prometheus 抓取配置参考：

```yaml
scrape_configs:
  - job_name: 'betterautosave'
    scrape_interval: 15s
    static_configs:
      - targets: ['<server-ip>:9450']
```

### `/betterautosave hottest-chunks` 命令

按保存延迟从高到低列出最慢的区块，默认显示前 10 个，可指定数量（最多 50）：

```
/betterautosave hottest-chunks 20
```

输出示例：

```
hottest-chunks (top 10 by window p99, window=100, tracking=8421/10000):
rank  pos              dim                          count   p99        max        last
1     [   12,  -34]    minecraft:overworld          100     12.34ms    45.67ms    3s ago
2     ...
```

高耗时区块通常出现在方块实体密集的位置——大型自动化农场、mod 商店面板、复杂红石装置等。定位到具体区块之后，就可以有针对性地调整。

### 新增配置

```toml
[prometheus]
enabled = false
bindAddress = "0.0.0.0"
port = 9450

[hottestChunks]
windowSize = 100       # 每个区块保留的采样次数，用于计算 p99
trackLimit = 10000     # 最多追踪多少个区块，超出后淘汰最久未保存的
```

### 升级

替换 jar 即可。配置文件会自动追加新增段，已有设置不变。

### 安全提示

Prometheus 端口默认监听 `0.0.0.0`。如果服务器在公网（云服务器 / VPS），请用防火墙限制 `9450` 端口的访问，或者把 `bindAddress` 改成 `127.0.0.1` 只允许本机连接。指标数据本身不包含玩家隐私，但会暴露服务器的活动模式。
