## v0.9.0 — 工具化与监控

BAS 把 chunk save / entity save / SavedData 三条主线程同步路径异步化已经稳了，这一版把诊断手段补齐：服主自助定位 mod / vanilla 瓶颈，不再依赖外接 spark profiler。

## 新增

### Prometheus metrics HTTP exporter

启用后在 `bindAddress:port/metrics` 暴露 17 counter / 5 gauge / 4 histogram，全部 `bas_` 前缀，零外部依赖（JDK 内置 HttpServer）。配 Grafana 可做生产长时间趋势监控，比单点 `/betterautosave debug` 快照诊断价值高得多。

启用方式（默认关闭）— `config/Shinoyuki-Optimize/shinoyuki_betterautosave/common.toml`:

```toml
[prometheus]
enabled = true
bindAddress = "0.0.0.0"
port = 9450
```

Prometheus scrape config:

```yaml
scrape_configs:
  - job_name: 'betterautosave'
    scrape_interval: 15s
    static_configs:
      - targets: ['<server-ip>:9450']
```

**安全提醒**: 默认 `0.0.0.0` 是为开箱即用。公网服 (云 / VPS) 必须用防火墙限制 9450 端口，或改 `127.0.0.1` 仅本地抓取。metrics 数据不直接敏感，但暴露世界活动模式。

### `/betterautosave hottest-chunks [count]`

列出滑动窗口内 worker NBT build p99 最高的 top N chunk（默认 10，最多 50），定位单点慢 chunk：

```
hottest-chunks (top 10 by window p99, window=100, tracking=8421/10000):
rank  pos              dim                          count   p99        max        last
1     [   12,  -34]    minecraft:overworld          100     12.34ms    45.67ms    3s ago
2     ...
```

应用场景：识别哪些 chunk 是 BlockEntity 多 / structure 复杂的 hot path，常见原因：
- 单 chunk 数百个 BlockEntity（大型自动化农场 / mod 商店面板）
- 复杂 structure NBT（大型船坞 / 地下城）

per-chunk 滑动窗口 (windowSize=100) + LinkedHashMap LRU eviction (trackLimit=10000)，内存 ~16 MB 可控。

## 配置项

```toml
[prometheus]
enabled = false                    # 默认 opt-in
bindAddress = "0.0.0.0"
port = 9450

[hottestChunks]
windowSize = 100                   # per-chunk 滑动窗口样本数
trackLimit = 10000                 # LRU 跟踪上限
```

## 升级 (从 v0.7.1)

直接替换 jar，配置文件会自动追加新增段，已有配置不动。无 breaking change。

## 数据安全

跟 v0.7.1 一致：所有提交不弱于 vanilla，关服走 vanilla 同步 flush 保证 100% 落盘，worker 异常自动 fallback。新增的 exporter / tracker 仅观测，不触碰存档路径。
